## Positive Observations

- The PARSE properties (Sections 1 and 1.1) are exceptionally complete. All seven VERDICT parsing cases from TSPEC-PARSE-01 through TSPEC-PARSE-04 are individually enumerated, including edge cases like intervening-text, truncated output, null/undefined input, extra/missing JSON keys, and negative integer values. Each maps to a specific FSPEC AT and TSPEC ref — this is exactly the right granularity for a contract-level property set.
- The LOOP properties (Section 2) correctly capture all eight loop-control behaviors including the dual-crash case (PROP-LOOP-08: two warning logs, one optimizer call, no halt). The POSTMORTEM prompt content check (PROP-LOOP-11) is a well-targeted enforcement mechanism for the AT-PM-F03 requirement.
- PROP-ENTRY-06 ("must use the guard `agent()` call, NOT `fs.existsSync`") is a technically precise property that directly enforces DEC-ODW-03's runtime constraint. This is the right type of Contract property.
- The COMPAT properties (Section 7) are well-scoped: checking unmodified baseline files with an exact file-existence test is machine-verifiable and immune to implementation drift.
- Section 10 Notes are honest about coverage boundaries (REQ-OBS-02, REQ-NFR-01 analytical verification, AT-HARVEST-03 origin). The transparency helps future reviewers understand the deliberate gaps.
- The Coverage Matrix (Section 8) is complete for all requirements except REQ-OBS-02, where the gap is documented.

## Recommendation

**Needs revision**

Three High findings require resolution before this PROPERTIES document can gate implementation:

1. **F-01 (High):** Add a `pipelineWiring.test.js` entry to the Section 9 Test File Index with the Phase 4 properties it covers. Either promote the REQ-OBS-02 `FinalReport` shape assertion and the Phase D conditional wiring into formal PROPERTIES entries (in Section 4 or a new Section 4.5), or explicitly document them as in-scope for `pipelineWiring.test.js` within the PROPERTIES table with the test level marked. Add REQ-OBS-02 to the Coverage Matrix.

2. **F-02 (High):** Either (a) reclassify PROP-IMPL-08 as requiring a defined git integration test fixture and document that fixture in the PLAN or TSPEC (specifying temporary-repository setup/teardown or shell-mock approach), or (b) split PROP-IMPL-08 into a unit-level property that mocks the shell layer and a separate integration property that specifies the test fixture. The current designation of "Integration" level in a unit test file is infeasible as written.

3. **F-03 (High):** Add a new Contract property (e.g., PROP-ENTRY-07 or a new Section 3.1) asserting that `pdlc/workflows/__tests__/helpers/guardAgentDouble.js` exists, exports `createGuardAgentDouble({ ok, reason })`, and is imported at every guard-agent test site (PROP-ENTRY-02, PROP-ENTRY-04, PROP-ENTRY-06, PROP-LOOP-06). Assign this property to `orchestrate-dev.test.js` at Unit level. Without this machine-verifiable enforcement, DEC-ODW-03's canonical-double requirement remains convention-only.

F-04, F-05, and F-06 (Medium) should also be addressed: F-04 by specifying a runtime-cache mock contract in the PLAN; F-05 by either re-labeling PROP-IMPL-01 as a static analysis check or specifying a call-order instrumentation approach; F-06 by adding the `{blocked-file-path}` extraction algorithm to TSPEC-HARVEST-04.
