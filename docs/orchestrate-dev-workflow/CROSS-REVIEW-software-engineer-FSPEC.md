## Findings

| F-01 | High | Local | **`reviewLoop` state machine is incomplete: §1 never specifies which reviewer skills and optimizer skill are used for each phase.** Section 1.2 says "both reviewer agents are dispatched in parallel" and the `reviewers` parameter is an `array[skill]`, but the FSPEC never provides a lookup table mapping each phase label (`R`, `F`, `T`, `P`, `PR`, `CR`) to its specific reviewer pair and optimizer skill. The caller must construct these values, but the FSPEC is the behavioral specification — an implementor of the script cannot determine the correct phase-to-reviewer mapping from §1 alone. The SKILL.md Reviewer Assignment Matrix exists, but the FSPEC must be self-contained on this behavioral rule. Without this table the `reviewLoop` construct cannot be implemented and tested independently of the SKILL.md prose runbook, defeating the purpose of the FSPEC. §4.8 partially addresses Phase CR, but R, F, T, P, and PR are unspecified. | §1.2, §6.2 |
| F-02 | High | Local | **VERDICT and `DECISIONS_WARRANTED` parsing algorithms are ambiguous about iteration strategy: the regex `/^…$/` is applied to a multi-line string but the strategy (split-and-iterate vs. multiline regex with `m` flag) is unspecified.** §2.2 step 1 says "Find the last occurrence of a line matching `/^VERDICT:\s+(.+)$/`" and §3.1 says "finds the last occurrence of a line matching `/^DECISIONS_WARRANTED:\s+(true\|false)$/i`". On a multi-line agent result, applying a regex with `^`/`$` anchors requires either (a) splitting the result into lines and iterating, or (b) using a regex engine's multiline mode. These produce different behavior: without the `m` flag, `^` and `$` match start/end of the full string, not individual lines. The "last occurrence" requirement is also only meaningful if lines are iterated — a non-multiline regex either matches the full string or doesn't. The algorithm must explicitly specify: split the result on newlines, iterate lines, track the last line matching the pattern. This is a normative parsing algorithm and must be deterministic and unambiguous for independent implementation. | §2.2 step 1, §3.1 |
| F-03 | High | Local | **Worktree merge strategy for implementation batches is unspecified: §4.5 step 4 says worktrees "are merged back" but the mechanism, strategy, and conflict handling are absent.** (a) Merge mechanism: does the script call a runtime `merge()` primitive, issue `git merge` via a Bash call, or does `isolation: "worktree"` automatically merge worktrees on agent completion? None of the confirmed runtime primitives (`agent()`, `parallel()`, `pipeline()`, `phase()`, `log()`) include a merge operation. (b) Merge strategy: fast-forward, squash, merge commit? (c) Conflict path: the SKILL.md error table includes "Merge conflict in worktree → Abort merge, halt, report conflicting files" but this case is completely absent from §7 (Cross-Cutting Error Flows). The §4.6 test gate depends on all batch agents having merged successfully — but there is no spec for what the script does if a merge fails before the test gate runs. The TSPEC cannot safely invent conflict handling without FSPEC guidance. | §4.5 step 4, §7 |
| F-04 | Medium | Local | **§4.6 test gate signal detection is underspecified: the script "inspects each agent's result for test failure signals" but no signal format is defined.** A batch "fails if any agent reports test failure or a non-zero exit code" — but what exactly constitutes a "test failure signal" in an `se-implement` agent's result string? `se-implement` agents produce prose output, not structured exit codes. Is there a structured return trailer analogous to the VERDICT trailer? If the script must parse free-form prose this is non-deterministic. If a new structured signal is required from `se-implement`, that is a behavioral change to a worker skill not mentioned in the FSPEC or REQ. This is the same class of problem that F-01 in the REQ review identified for the VERDICT contract. The consuming script needs a deterministic, machine-readable test-result signal; the FSPEC must specify its format or explicitly call out the prose-parsing approach and its limitations. | §4.6 |
| F-05 | Medium | Local | **Creator agent invocations are entirely unspecified for phases F, T, D, P, PR.** §6.2 types these phases as "Create then `reviewLoop`" but the FSPEC contains no section specifying the creator agent call: which skill, what inputs it receives, what output path it must produce, and what happens if the creator agent fails before producing the document. §7.2 handles the case where the document is missing when `reviewLoop` is called, but it frames this as an indirect detection ("this typically indicates the creator agent failed") rather than a direct failure path. There is no spec for what the creator invocation looks like, what inputs it receives, or how the script distinguishes "creator succeeded but document is missing" from "creator never ran." The TSPEC must specify these agent calls, but they are behavioral rules the FSPEC owns. | §1.1, §6.2, §7.2 |
| F-06 | Medium | Local | **Resume detection algorithm in §1.7 is unimplementable: the script must "detect it is continuing an in-progress loop" but no runtime API for cache-state querying is specified.** The condition "cached results exist for some but not all agents in the current iteration" implies the script can query the runtime cache state before invoking `agent()`. However, none of the confirmed runtime primitives (`agent()`, `parallel()`, `pipeline()`, `phase()`, `log()`) expose a cache-query capability. If no such API exists, the detection algorithm is unimplementable as written. The FSPEC must either (a) identify the runtime API used to detect cached-vs-uncached agent calls, or (b) state that resume detection is best-effort — the log is emitted unconditionally at the start of every `reviewLoop` call except when the iteration counter is 1 (i.e., no special cache-state detection required). | §1.7, REQ-PIPELINE-03 |
| F-07 | Medium | Local | **Phase CR `reviewLoop` call breaks the §1.1 `docPath` precondition: Phase CR has no single document path but `reviewLoop` accepts `docPath` as a single file path.** §4.8 says the "Document context: REQ acceptance criteria + PROPERTIES + current feature branch" — but `reviewLoop`'s `docPath` parameter (§1.1) is typed as a string (a file path), and the §1.1 entry precondition checks that "the upstream document at `docPath` exists." No single file represents "the implementation." The FSPEC must specify: (a) what value the script passes as `docPath` for Phase CR (e.g., the feature branch directory path, or a synthesized path), (b) how reviewers are given a multi-document scope rather than a single file, and (c) how §7.2 ("document does not exist") behaves when `docPath` is not a conventional artifact path. Without this, the `reviewLoop` abstraction is not closed over Phase CR. | §4.8, §1.1 |
| F-08 | Medium | Cross-Feature | **Phase H gating prerequisite in §5.1 is a conditional skip with no specified detection mechanism or scripted representation.** The prerequisite "Do not enable Phase H until the feature-branch-consistency fix has landed" is an operational note carried from the SKILL.md, but the FSPEC must specify how this translates into script behavior: is it a `const HARVEST_ENABLED = false` flag in the script, a runtime branch check, or something else? Without a specified mechanism, the TSPEC author must invent the implementation. Additionally, the FSPEC does not specify what the workflow emits or does when Phase H is skipped due to this prerequisite (vs. Phase H completing normally) — there is no `log()` message or `/workflows` representation defined for the "prerequisite not met" skip path, unlike Phase D's skip path which is fully specified in §3.2. This is Cross-Feature because the worktree-artifact-visibility constraint will affect every future workflow that uses `isolation: "worktree"` with downstream file reads. | §5.1 |
| F-09 | Low | Local | **Iteration counter increment timing is ambiguous between §1.5, §1.6, and §1.8.** §1.5 step 6 says "The iteration counter increments and the loop returns to §1.2." §1.6 says "The counter increments after every optimizer invocation … before the next parallel dispatch." §1.8 says "If the iteration counter would exceed 5 and the gate state is still FAIL, the script does not invoke the optimizer again." The phrase "would exceed 5" implies the counter increments before the cap check, but §1.5 places the increment after the optimizer call. If the counter increments at the end of iteration 5's optimizer invocation to become 6, then "would exceed 5" is checked as "counter == 6." If the counter increments before the optimizer check, then after iteration 4's optimizer the counter becomes 5, iteration 5 runs, and the check is "counter == 5 AND FAIL → skip optimizer." The two readings produce the same outcome (POSTMORTEM triggered after 5 iterations) but differ in whether the counter value at POSTMORTEM time is 5 or 6, which affects the POSTMORTEM content and the `log("Resuming from iteration N")` value on resume. | §1.5, §1.6, §1.8 |
| F-10 | Low | Local | **§2.2 step 6 JSON validation says "exactly the keys `high`, `medium`, `low`" but does not specify that this is a set comparison, not an ordered comparison.** `JSON.parse` does not guarantee key order. An implementor who writes `Object.keys(json).join(',') === 'high,medium,low'` would produce a parser that rejects valid `{"medium":0,"high":0,"low":0}` objects. Since §2 is labeled as the normative parsing algorithm, it should explicitly state: the check is that the parsed object's key set equals `{"high","medium","low"}` (set equality, order-independent). | §2.2 step 6 |
| F-11 | Low | Local | **§4.3 sub-batch ordering invariant is vacuous and the sequential-vs-parallel question for sub-batches is unanswered.** All tasks in a topological batch are by definition dependency-independent from each other. "Preserving topological ordering within each sub-batch" is a vacuous constraint — any ordering preserves it. The FSPEC should clarify: (a) sub-batch ordering is by document order in the PLAN task table (deterministic), and (b) whether sub-batch 1a must complete before sub-batch 1b starts (sequential sub-batches) or whether 1a and 1b can run in parallel as one combined batch (since all 8 tasks in the example are equally ready). The example in §4.3 ("both run before any dependent tasks") is ambiguous on this point. The implementation choice has a material impact on the agent fan-out rate and thus on the REQ-NFR-01 concurrency cap compliance. | §4.3 |
| F-12 | Low | Local | **§3.1 does not specify whether the `DECISIONS_WARRANTED` trailer is injected by the workflow script at invocation time or baked into the `se-author` SKILL.md.** REQ-COMPAT-03 states worker skills are behaviorally unchanged except for the verdict trailer on the three reviewer skills. Baking `DECISIONS_WARRANTED` into `se-author`'s SKILL.md would constitute a behavioral change to a worker skill beyond the agreed scope of changes. The FSPEC should state that this trailer is a workflow-script-injected instruction appended to the agent call (not a permanent `se-author` SKILL.md change), matching the resolution approach for VERDICT trailer placement. This closes OQ-03 for the TSPEC author. | §3.1, REQ-COMPAT-03 |
| F-13 | Low | Local | **Error message placeholder notation is inconsistent across §2.3, §7.4, and §7.5, making it impossible to derive authoritative test assertion strings from the FSPEC alone.** §2.3 uses `<skill-name>` with angle brackets and provides examples (`se-review`, `te-review`, `pm-review`). §7.4 uses `{optimizer-skill}` with curly braces. §7.5 uses prose ("the hook, the blocked tool call, and the file path involved") with no format at all. The acceptance tests in §8 reference these message formats (AT-LOOP-04, AT-HARVEST-02, AT-ENTRY-01/02) but a test implementor cannot derive a single authoritative regex or literal string from the inconsistently-specified format. ## Questions

| ID | Question |
|----|---------|
| Q-01 | §4.5 specifies `isolation: "worktree"` for `se-implement` dispatch. The current SKILL.md uses `isolation: "worktree"` for reviewer agents too. Does the workflow script use `isolation: "worktree"` for reviewer agents in `reviewLoop`, or direct-commit? The FSPEC is silent on reviewer isolation mode. If reviewers use direct-commit and both write cross-review files in parallel, there is a potential write conflict on the feature branch. If they use worktrees, the merge strategy gap in F-03 applies to review phases as well. |
| Q-02 | §4.7 Phase PT says "The script invokes a single `se-implement` agent to implement any PROPERTIES tests not yet covered." How does the script determine which tests are "not yet covered" — is this a prompt instruction to the agent (it reads PROPERTIES and decides), or does the script perform a static analysis? |
| Q-03 | §8 acceptance tests label AT-VERDICT-01 through AT-VERDICT-04 as "parser unit" tests. Is the intended test architecture a unit test on an exported `parseVerdict(result: string)` function in `orchestrate-dev.js`, or an integration test on `reviewLoop` with a stubbed agent? The answer determines the TSPEC's module boundary decisions for the parser. |
| Q-04 | §1.9 step 1 specifies the POSTMORTEM agent receives a quoted instruction string beginning "Write `docs/{featureName}/POSTMORTEM-…`". Does the workflow runtime's `agent()` call support arbitrary instruction strings as the skill content, or must it reference a named skill? If arbitrary instructions are not supported, the POSTMORTEM writing must be routed through a named skill with a parameter. |

## Positive Observations

- The VERDICT trailer contract in §2 is detailed, normative, and deterministic. The use-last-occurrence strategy (§2.2 step 1) correctly handles agents that mention VERDICT in prose before the final trailer. The exhaustive fallback condition table in §2.3 covers every parser failure mode including the JSON-malformed case that straddles the VERDICT-present/JSON-invalid boundary — this was an open gap in the REQ that the FSPEC closes cleanly.
- The harvest ordering guarantee in §5.2 is well-structured: commit-push-before-delete ordering combined with the guard hook as a second enforcement layer creates a defense-in-depth approach that is both testable (AT-HARVEST-01, AT-HARVEST-02) and auditable via git log.
- The 5-iteration cap in §1.8 and §1.9 is well-specified: the POSTMORTEM agent receives an explicit instruction block, the POSTMORTEM-agent-failure path is handled (§1.9 step 2), and the final report content for non-convergence is enumerated (§1.9 step 4). This is a significant improvement over the SKILL.md which treats this as a prose step.
- The DECISIONS conditional in §3 resolves REQ review F-15: the `DECISIONS_WARRANTED` trailer format, case-insensitive parse, and safe-default (`true`) for missing/malformed values are all specified. The `/workflows` representation for both skip and full paths is explicitly stated — useful for independent test assertions.
- Pipeline entry validation in §6.1 provides exact error message strings for all four failure modes, making AT-ENTRY-01 and AT-ENTRY-02 directly implementable as string-match assertions without TSPEC elaboration.
- §7 correctly distinguishes halt conditions (§7.2, §7.4) from advisory-only behaviors (§7.6, §7.7), with `check-scope-field` hook non-blocking behavior and `nudge-consolidation` session-scoping correctly documented.

## Recommendation

**Needs revision**

Three High findings and four Medium findings require resolution before the TSPEC can be authored:

- **F-01 (High):** Add a phase-to-reviewer-and-optimizer lookup table to the FSPEC covering all phases. The SKILL.md Reviewer Assignment Matrix must be reproduced here as a first-class FSPEC behavioral rule.
- **F-02 (High):** Specify the parsing strategy for both the VERDICT regex (§2.2) and `DECISIONS_WARRANTED` regex (§3.1) as split-and-iterate (split result on newlines, iterate, track last match), not multiline regex, to ensure determinism across JavaScript regex implementations.
- **F-03 (High):** Add a §4.5 sub-section (or §7 entry) specifying the worktree merge mechanism (runtime automatic vs. explicit script call), merge strategy (fast-forward/squash/merge-commit), and conflict handling path.
- **F-04 (Medium):** Specify the `se-implement` test-result return signal format — either define a structured trailer or specify the exact string pattern the script uses to detect test failure.
- **F-05 (Medium):** Add creator agent invocation specs for phases F, T, D, P, PR: which skill, required inputs, expected output path, and direct failure path.
- **F-06 (Medium):** Specify the resume detection mechanism or explicitly state it is best-effort (no cache-state query API required; log emitted based on iteration counter state alone).
- **F-07 (Medium):** Specify what value is passed as `docPath` for Phase CR's `reviewLoop` call and how reviewers receive a multi-document review scope.

F-08 (Medium, Cross-Feature) should be addressed by specifying whether the Phase H prerequisite is a compile-time constant or runtime check, and what the workflow emits when Phase H is skipped due to the prerequisite.

Low findings F-09 through F-13 do not block TSPEC authoring individually but should be resolved to avoid pushing implementation ambiguity into the TSPEC.
