## Questions

| ID | Question |

| Q-01 | §1.7 states `log("Resuming from iteration N")` uses N = "the iteration that was active when the run was interrupted, not the next iteration to execute." Does this log fire at the start of the resumed agent dispatch (before the first re-executed call), or at the point the runtime hands back control to the script? The placement in the `/workflows` view relative to re-dispatched agent calls depends on this timing. |
| Q-02 | §4.6 specifies batch failure detection via "non-zero exit code" from the `se-implement` agent, but §2.2 describes agent results as strings inspected for VERDICT trailers. Do `se-implement` agents produce a structured result with an exit code, or does "non-zero exit code" refer to a signal embedded in the result string? The asymmetry between review-agent and implementation-agent failure signaling should be resolved in the FSPEC or explicitly deferred to the TSPEC. |
| Q-03 | §4.8 states the Phase CR optimizer is `se-author` acting to "address implementation review feedback, not a document author." This invocation role is distinct from every other `se-author` invocation (which authors TSPEC, PLAN, or DECISIONS documents). Is there a separate skill configuration for the CR optimizer invocation, or is `se-author` invoked with a different task instruction? If the same skill is used, the TSPEC will need to specify how `se-author` is instructed to review code rather than author a document. |
| Q-04 | §5.1 includes a "Gating prerequisite" note that Phase H should not be enabled until a feature-branch-consistency fix lands. Is this prerequisite tracked as a named requirement or issue? If Phase H is always skipped at initial ship, AT-HARVEST-01 and AT-HARVEST-02 cannot be validated in the first release. The PROPERTIES document should classify these ATs as deferred; the FSPEC should explicitly state the prerequisite status to guide that classification. |

## Positive Observations

- The VERDICT parsing algorithm in §2 is exceptionally well-specified. The last-occurrence rule for disambiguating intermediate VERDICT mentions, the explicit prohibition on blank lines between VERDICT and JSON, and the exhaustive enumeration of fallback triggers in §2.3 provide enough detail to write a complete unit test suite for the parser without further clarification. This is the precision standard every FSPEC behavioral contract should meet.
- The DECISIONS conditional in §3 resolves the REQ-level ambiguity from TE-REQ-v2 F-15 and SE-REQ-v2 F-15 cleanly. The `DECISIONS_WARRANTED:` trailer format is normatively specified with a regex, case-insensitive parsing is stated, the missing-field safe default (true) and its exact log message are given, and the observable differences between skip and full paths are tabulated in §3.4. The conservative default (true) is the correct choice for testability.
- §1.9's POSTMORTEM branch is the most fully-specified non-convergence path in this codebase. Enumerating both the POSTMORTEM success path and the POSTMORTEM agent failure path — with the exact warning message for the failure case — converts a previously implicit failure mode into a testable contract. The only gap is that the failure path (F-01) lacks an AT.
- The traceability table in §9 links every FSPEC section to its parent REQ requirements. Every section has at least one REQ link, no spurious cross-references, and the mapping is accurate against the REQ.
- §7.4's optimizer failure error message (`"Error: optimizer agent {optimizer-skill} failed during phase {phase}, iteration {N} — pipeline halted. Document at {docPath} may be in an inconsistent state."`) is precise enough to assert against without further specification. The rationale for no-retry is stated and sound. The same precision applied to all halt-path error messages in this FSPEC.
- AT-DECISIONS-02 correctly tests the missing-field-defaults-to-true path separately from an explicit-true path, demonstrating awareness that they exercise different parser branches. The Who/Given/When/Then format is consistently applied throughout §8.

## Recommendation

**Needs revision**

Three High findings (F-01, F-02, F-03) require new or amended acceptance tests before the PROPERTIES document can be authored against this FSPEC. The gaps are: a named error path with no AT (POSTMORTEM agent failure, F-01), a named mid-pipeline halt path with no AT (document missing before `reviewLoop`, F-02), and a truncated-output parsing edge case not reachable by any existing AT (VERDICT line present but no following line, F-03). Authoring PROPERTIES against an FSPEC missing these ATs would produce a PROPERTIES document with equivalent coverage gaps.

The Medium findings (F-04 through F-09) add missing ATs for explicitly named behavioral branches: dual-reviewer crash, resume iteration-counter semantics, explicit `DECISIONS_WARRANTED: true`, optimizer failure halt, and DAG inconsistency override. These must be addressed before PROPERTIES authoring.

Required changes in priority order:

1. **F-01:** Add AT-LOOP-05: POSTMORTEM agent itself fails — assert warning note in final report, workflow still halts, no retry.
2. **F-02:** Add AT-LOOP-06: `reviewLoop` called with non-existent `docPath` — assert halt with exact error message, no reviewer agents dispatched.
3. **F-03:** Add AT-VERDICT-05: VERDICT line is the last line of agent output with no subsequent line — assert fallback triggered, treated as `Needs revision`, warning logged.
4. **F-04:** Add AT-LOOP-07: Both reviewers crash in the same iteration — assert two warning logs emitted, gate FAIL, optimizer invoked once, iteration counter increments.
5. **F-05:** Add AT-LOOP-08: Run interrupted mid-iteration 3, resumed — assert `log("Resuming from iteration 3")`, not `Resuming from iteration 4`; iteration cap counter not reset to 1.
6. **F-06:** Add AT-DECISIONS-03: TSPEC optimizer explicitly returns `DECISIONS_WARRANTED: true` — assert `reviewLoop` for Phase D entered without the absent-field warning log.
7. **F-07:** Add AT-ERROR-01: Optimizer agent fails during phase R, iteration 2 — assert pipeline halts, exact error message format, no retry.
8. **F-08:** Add a unit-level AT for §4.4: assert `log()` call for batch plan precedes the first `agent()` call for any `se-implement` task (call-order assertion, not UI observation).
9. **F-09:** Add AT-IMPL-03: PLAN has a batch-2 task depending on another batch-2 task — assert script overrides to produce batch-1 (prerequisite) + batch-2 (dependent).

The Low findings (F-10 through F-18) do not individually block revision but should be resolved in the same pass.
