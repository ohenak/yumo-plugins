| F-01 | Medium | Local | **CF-7's bespoke-row convention re-opens the silent coverage-drop hole the § 6.3 self-audit exists to close.** A row kept in `MATRIX` with `bespoke: true` is filtered out of `it.each`; its coverage then depends entirely on a hand-written dedicated `it()` block that **nothing audits**. Delete (or forget to write) that block and all three meta-tests still pass — meta-test 1 sees the id in `MATRIX`, the length check sees no duplicate — while the row executes **zero** times: exactly the "deleted row silently drops coverage" failure the self-audit table promises to prevent, and a silent breach of REQ-GUARD-05's exactly-one obligation on the bespoke subset (this feature's review lineage rates unbound/unallocated rows Medium — TE TSPEC F-01, PM TSPEC F-06). The symmetric direction (a dedicated block for an *unmarked* row → double execution) is equally un-audited. Fix inside CF-7: dedicated blocks are generated mechanically from the marked entries via an `id → orchestration function` dispatch map, and meta-test 1 gains a fourth conjunct (or a meta-test 4): the dispatch map's key set deep-equals the set of `bespoke: true` ids in `MATRIX` — closing both the zero-execution and double-execution directions | CF-7; TSPEC § 6.3 self-audit |
| F-02 | Low | Local | **CF-3's `NOT_COMMITTED` → `commit` conjunct is unfalsifiable for the regression it exists to catch.** The § 3.7 template's state description "LEARNINGS-{feature}.md **is not committed** for docs/{feature}/" itself contains the substring `commit`, so a catalog regression that drops the "then commit (and push)" instruction — the REQ-GUARD-07 required content ("the instruction to run `/pdlc:harvest-learnings` **and commit**") the conjunct protects — still passes every `NOT_COMMITTED` row. The added conjunct provides zero protection as specified. Assert a discriminating substring instead: `then commit` (or `commit (and push)`). The other two CF-3 conjuncts (`ailing closed`, `interpreter`) are discriminating and fine as written | CF-3; TSPEC § 3.7 / § 6.3; REQ-GUARD-07 |

- **DEC-04 is exemplary evidence discipline for a version-sensitive fixture**: the rejected corrupt-ref alternative is recorded *with* its empirical failure mode (clean absence, G5 routing), the adopted fixture matches C18 exactly, and the builder self-check turns the loose-object assumption into a loud failure — the re-evaluation trigger is mechanically observable rather than aspirational.
- **CF-6 fully closes TE F2-04**: the floor is named, hard-bounded below at 1, and asserted by the jest binding against printed output — the emptied-table and gate-bug-exit-0 cases both now fail.
- **The decisions are matrix-faithful**: every pin cited across DEC-05–DEC-10 (M87–M90, M82, M75/M76/M83/M85/M86, M17/M64–M66/M80, M42–M43/M68–M72, M15/M56/M62/M67, M20/M73) was checked against REQ v1.7 and none contradicts a row, the § 6.3 allocation, or the M33 re-run enumeration.
- **The generative-testing rejection in DEC-01 is a justified property-strategy exemption, not an evasion**: the constraint (stdlib-only floor; single-file deployable) is real, and the P-D1/P-QUOTE tables plus the self-test corpus state the invariants (D1/NFR-01, quoting invariance) over an enumerable corner space deterministically.
- **CF-1's "pick one, don't stack both silently" clause** is good test-determinism hygiene — and, usefully, both sanctioned mechanisms are observationally equivalent, so a single oracle (F-03) pins either.
