## Positive Observations

- The REQ correctly identifies real, verified defects: I confirmed `guard-harvest-before-delete.sh:35,37` only matches literal tokens + `rm`/`unlink`/`git rm`, `:52` checks disk not commit state, and `check-scope-field.sh:41` matches the bare substring `scope` case-insensitively ("telescope" indeed false-passes).
- Who/Given/When/Then ACs with explicit exit-code expectations (exit 2 / exit 0) are directly executable in the existing `runHookScript` harness pattern — the AC *format* is right; the matrix is just incomplete.
- Explicit negative requirement (REQ-GUARD-NFR-01) and explicit residual-risk carve-out (Write-tool truncation) show the right fail-closed thinking; the out-of-scope list is honest.
- Test requirement (REQ-GUARD-05) is P0, not an afterthought — correct for a security-posture hook.

## Recommendation

**Needs revision**

Required changes, in priority order:
1. Define the indeterminate-input set and give the fail-closed clause its own block/allow ACs (F-01).
2. Specify guard behavior for non-repo, no-upstream, detached-HEAD, and stale-ref git states, and require hermetic bare-remote test fixtures (F-02).
3. Declare expected behavior for the interpreter-missing / unparseable-stdin paths and either extend the verb matrix to interpreter/pipe wrappers or state they are accepted residual risk (F-03).
4. Pin the refusal message oracles, `mv` path-resolution rules, compound-command ACs, Scope-pattern regexes, false-block allow ACs, and LEARNINGS name derivation (F-04–F-09).
