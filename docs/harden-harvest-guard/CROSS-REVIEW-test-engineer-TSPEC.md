| F-02 | Medium | Local | **Degraded-row harness cannot execute as specified: the restricted `PATH` prevents spawning `bash` itself.** On POSIX, Node's `spawnSync` resolves the executable via the *child* env's `PATH` (libuv installs `options.env` as `environ` before `execvp`). Empirically verified in this repo: `spawnSync("bash", …, {env: {PATH: <empty mkdtemp dir>}})` → `error: ENOENT`, `status: null` (→ `exitCode: -1` in `runHookScript`). So all seven degraded rows (M42–M43, M68–M72) fail at spawn — the wrapper never runs, `expectBlock`/`expectAllow` fail on -1, and REQ-GUARD-06's P0 degraded policy has no working test design. Fix: `degradedEnv()` rows must spawn bash by **absolute path** (resolve `command -v bash` under the parent PATH once, then `spawnSync("/bin/bash", [scriptPath], …)` with the restricted env) — the script is invoked as an argument, so no shebang resolution is needed. Note the failure is loud, not a false green, but the specified architecture is non-functional for these rows. | § 6.2 `degradedEnv()`, § 6.1 |

## Positive Observations

- **Every existing-code claim (C1–C16) checked out** against the actual scripts, harness, package.json, and hooks.json — line citations accurate, including the subtle C11 (undefined-valued env entries dropped by `spawnSync`).
- **§ 3.1's builtins-only stdin read is an excellent catch**: the current `input="$(cat)"` would silently yield `""` under the restricted-PATH fixture and M42 would false-allow — the TSPEC found and closed a degraded-mode hole in its own test design's blind spot.
- **Step-2b hoist re-traces independently verified correct**: M82 (indeterminate destination → 2b requires static, unaffected, still D3/INDETERMINATE), M74 (step 2 fires first, unaffected), M84 (blocks at 2b with the same code as the FSPEC step-4 routing), M11/M52 (destinations name no existing file / are existing directories → 2b excluded; step-5 allows). No block/allow class changes anywhere.
- **T-01–T-03 are genuinely discriminating**, each killing a named wrong implementation (step-3-first → INDETERMINATE on T-01; guarded-source-exempting → step-5 ALLOW on T-02; literal-destination-only → fall-through ALLOW on T-03), and their G6-allow expectations are correctly derived.
- **Hermeticity design is sound**: local bare origin, `git update-ref` stale-ref construction for G10, unreachable origin via `set-url` to a nonexistent path (fast, network-free), mkdtemp isolation, local identity config.
- **The `ls-tree` vs `cat-file -e` idiom analysis** (clean-empty absence vs `fatal:` diagnostics) correctly operationalizes the FSPEC failure-vs-absence boundary.
- The self-audit meta-test concept and the assertion-level PROP-COMPAT-05 migration map (rows 1–2) are exactly the right disciplines for a matrix-bound suite.

## Recommendation

**Needs revision**

Six Medium findings. What must change: (F-01) allocate M74/M84 base-state tests in § 6.3; (F-02) spawn bash by absolute path for degraded rows; (F-03) promote T-01–T-03 + step 2b upstream (REQ v1.7 / FSPEC v1.4) or drop the hoist; (F-04) add at least one property-based strategy for the parsing engine (black-box NFR-01 grammar and/or quoting-invariance properties suffice); (F-05) enumerate the full REQ-GUARD-07 per-reason substring assertions; (F-06) add a fixture that actually drives the git query-failure branch (corrupt object store), since the corrupt-ref fixture empirically classifies as clean absence under `show-ref --verify --quiet`.
