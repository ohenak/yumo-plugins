Through the `mv` flow: step 2 misses (source unguarded), step 3 misses (nothing indeterminate), step 4 misses (source not a guarded file) → fall-through ALLOW — the guarded artifact is destroyed with exit 0. Through general D2 (FSPEC-GUARD-01 / REQ-GUARD-01): `mv` is a defended deletion verb with a static operand resolving to a guarded file → D2(i) BLOCK `NOT_COMMITTED`. Two normative texts, two exit codes, no row: M11/M52 pin only that `mv` *destinations* are exempt from D2(ii)-style path tests when the source is guarded and the resulting path stays in-feature; nothing pins the unguarded-source case. The incoherence is sharpened by v1.2 itself: step 3's new rationale — "the move can still **overwrite (destroy) a guarded file**" — is the ground for blocking the *indeterminate* spelling (M82), yet the *statically provable* spelling of the same destruction allows: the guard would block when it cannot know and allow when it knows for certain. US-01 promises blocking for deletion "spelled with any defended deletion verb", the truncation analog `> docs/f/CROSS-REVIEW-x.md` blocks (M14), and the Residual Risk Register names every other accepted bypass (RR-W, RR-1–RR-5) — an unregistered allow inside the defended verb set is a hole by this REQ's own discipline. Fix (either way is a same-commit matrix amendment per TE F-09): **(a)** add a static-destination test to the `mv` flow — if the static resulting path resolves to an **existing** guarded file in an unverified guarded directory → BLOCK with the G-state reason code (the existence qualifier is what keeps M11/M52 allowed: their resulting paths name no existing file) — and pin it (e.g. M84: `mv /tmp/notes.md docs/f/CROSS-REVIEW-x.md` → BLOCK `NOT_COMMITTED`, in the G6 re-run set); or **(b)** register static-`mv`-overwrite as a named residual risk with an ALLOW row.
