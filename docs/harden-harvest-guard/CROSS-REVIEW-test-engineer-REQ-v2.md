| F2-01 | High | Local | **Quote semantics are internally contradictory, and the Canonical Matrix has no quoted-target row — a quotes-only bypass would pass all 43 rows green.** Parsing rule 1 says content inside single/double quotes "is data and is **never scanned for verbs or paths**"; rule 3 says operands of deletion verbs ARE inspected as candidate paths; rule 5 says an operand is static iff it contains "no **unquoted** `$`". Apply these literally to the most common agent spelling, `rm "docs/f/CROSS-REVIEW-x.md"`: rule 1 → quoted content never scanned as a path → ALLOW; rules 3+5 → operand with no `$` at all → static → resolves to a guarded file → D2 BLOCK. Two rules, two opposite outcomes. Same defect inverts M24: in `rm "$D"/*.md` the `$` is *inside double quotes*, so by rule 5's literal text the operand has "no unquoted `$`" → static — yet M24 mandates `INDETERMINATE`. So "unquoted" evidently means "not single-quoted / expansion-active", but the REQ never says so. Every AC and matrix row M01–M43 uses only unquoted targets, so an implementation that follows rule 1 literally (quoted paths invisible) passes the entire durable oracle while `rm "docs/f/CROSS-REVIEW-x.md"` and `rm 'docs/f/*.md'` sail through — precisely the class of bypass this feature exists to close. **Fix:** (a) state that shell-style tokenization with quote removal precedes operand classification — a quoted operand of a deletion verb is still a candidate path; rule 1's data exclusion applies to string arguments of non-deletion verbs, heredoc bodies, and quoted regions that do not form deletion-verb operands; (b) define "unquoted" in rule 5 as "not single-quoted" (double quotes do not suppress `$`/`` ` ``/`$(` expansion); (c) add matrix rows: `rm "docs/f/CROSS-REVIEW-x.md"` → BLOCK NOT_COMMITTED, `rm 'docs/f/CROSS-REVIEW-x.md'` → BLOCK NOT_COMMITTED, and `D="docs/f"; rm "$D"/*.md` → BLOCK INDETERMINATE (the double-quoted-assignment variant of M24, which also needs D3(a) to say whether a *double-quoted* `docs/` token satisfies the docs-reference condition). | REQ-GUARD-01 (parsing rules 1/3/5, D2/D3), Canonical Matrix |
| F2-02 | Medium | Local | **`mv` directory-destination semantics are still undecidable — the exact case raised in iteration-1 F-05.** For `mv docs/f/CROSS-REVIEW-x.md docs/f/archive/` (destination is an existing directory inside the feature dir), row 2 of the `mv` table requires "destination basename still matches `CROSS-REVIEW-*.md`". Reading A: destination basename is `archive` → no match → row 3 BLOCK ("pattern-destroying rename"). ## Recommendation

**Needs revision**

Required changes, in priority order:
1. Reconcile the quote semantics (parsing rule 1 vs rules 3/5), define "unquoted", and add quoted-target rows to the Canonical Matrix so a quotes-only bypass cannot pass the oracle green (F2-01).
2. Decide the `mv`-into-directory reading and the nested-subdirectory guarded-file/feature-derivation question; add the matrix row (F2-02).
3. Complete the oracle: add G5, G10-default-`false`, and fetch-failure rows so every normative behavior in REQ-GUARD-03 is bound to a test (F2-03).
4. Low: RR-4 register entry for the G2/G3 fallback; classify `1>` / `>|` (F2-04, F2-05).

The revision is close: v1.1 resolved all eleven iteration-1 findings and all three questions on their own terms, and the matrix-as-oracle structure is exactly right. The remaining defects are consistency holes *inside* the new machinery, not missing machinery.
