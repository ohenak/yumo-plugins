| F-01 | Medium | Local | **The `mv` flow's internal ordering contradicts BR-01-1's first-match-wins D2 → D3 and changes an observable reason code.** FSPEC-GUARD-02 evaluates step 2 ("source **or destination** is indeterminate → D3") *before* step 3 ("source is an unverified guarded directory or ancestor → BLOCK"). For `mv docs/f "$DEST"` (static source = the unverified guarded directory itself; indeterminate destination), step 2 fires first → BLOCK `INDETERMINATE`. Under BR-01-1 / REQ D-rule ordering, D2 evaluates first — the static source operand resolves to the unverified guarded directory (D2 iii) — → BLOCK with the G-state reason (`NOT_COMMITTED` in the default fixture), and the message tells the agent to harvest/commit rather than reporting an unresolvable operand. Both paths block, but REQ-GUARD-07 makes the reason code the test oracle, no matrix row pins the mixed static-guarded-source/indeterminate-destination case, and a TSPEC author cannot derive which code to emit: FSPEC-GUARD-01 BR-01-1 says "no other precedence exists" while FSPEC-GUARD-02 installs an mv-internal precedence in which indeterminacy trumps static-source knowledge. ## Positive Observations

- The six-flow decomposition plus matrix-by-reference is the right shape: behavior lives in the flows, the oracle lives in the REQ matrix, and the single FSPEC-added row (M66) is introduced with an explicit Given/When/Then and a G6 re-run extension rather than as a bare table line. The row-assignment partition over M01–M66/S01–S06 is exact (verified — no gaps, no double-ownership).
- All three carry-forwards are genuinely discharged with back-references, and CF-1's correction is shell-accurate — the FSPEC is more correct than the REQ it supersedes on that point, with the supersession stated rather than silent.
- The reason-code mapping table in FSPEC-GUARD-06 (decision path → code) is the right mechanism for REQ-GUARD-07's "selected mechanically" requirement; F-02 is a completeness gap in the table, not a design flaw.
- Existing-code claims are accurate, including the load-bearing M66 hermeticity claim (`runHookScript` already exposes both spawn `cwd` and stdin content — `hookCompatibility.test.js:47-52`).
- BR-02-2 preserving the `git clean` rationale for TSPEC, and BR-03-1 restating the env-var threshold ownership, both prevent the classic loss-in-translation failure where downstream documents treat inherited decisions as arbitrary.

## Recommendation

**Needs revision**

Three Medium findings must be resolved before TSPEC authoring: F-01 (mv flow ordering vs BR-01-1 — pins an observable reason code), F-02 (G1 eager/lazy precedence and the double-matching GUARD-06 rows), and F-03 (recursion-cap latitude contradicting the P0 no-false-block guarantee).
